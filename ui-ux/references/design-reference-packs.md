# Design Reference Packs

Major design systems are reference material, not skins to clone. Pick 2–4 that match the product temperament, then borrow principles and patterns — not visual identity.

## How to choose

1. Identify the product temperament from `SKILL.md`.
2. Pick the matching section below.
3. Select 2–4 references from that section.
4. Fill in the output template at the bottom.

---

## Developer Precision

For APIs, SDKs, developer consoles, protocol tools, infrastructure, and code-adjacent products.

References:

- GitHub Primer
- Stripe Docs / Stripe Apps
- GitLab Pajamas
- Vercel / Linear (as product references)

Borrow:

- Compact hierarchy.
- Code / ID / copy affordances.
- Clear, actionable error messages.
- Trustworthy, readable typography.
- Low-noise layouts that prioritize task over atmosphere.

Avoid:

- Landing-page hero on tool pages.
- Decorative gradients that block task flow.
- Raw JSON as the only result explanation.

---

## AI Native Future

For AI agents, bot studios, workflow builders, knowledge products, and creative tools.

References:

- OpenAI / ChatGPT product surfaces (general product inspiration).
- Anthropic Console-style clarity.
- Linear / Raycast / Notion AI / Vercel-style modern product feel.

Borrow:

- Calm, futuristic feel.
- Fluid but purposeful interactions.
- Workspace + preview / test panel layouts.
- Modular composition.
- Intelligent suggestions surfaced contextually.

Avoid:

- Generic purple-blue AI glass template.
- Retro aesthetic unless product concept deliberately supports it.
- Enterprise CRUD look for a novel creative product.

---

## Enterprise Ops

For operations, diagnostics, monitoring, admin control, and complex state management.

References:

- IBM Carbon
- Atlassian Design System
- GitLab Pajamas
- Datadog / Grafana-like operational patterns

Borrow:

- State-first layout.
- Issue queues and severity levels.
- High-density tables.
- Diagnostic affordances.
- Accessibility.
- Clear action hierarchy.

Avoid:

- Full IBM / enterprise-blue skin unless explicitly desired.
- Dashboard charts that do not answer operational questions.

---

## Commerce Trust

For licensing, billing, authorization, deployment distribution, and high-trust workflows.

References:

- Stripe
- Shopify Polaris
- GitLab Pajamas
- Atlassian

Borrow:

- Precision and auditability.
- Cautious destructive-action patterns.
- Clear, unambiguous forms.
- Evidence trails and operation logs.
- Trust-building copy.

Avoid:

- Playful or gamified surfaces for risky financial or authorization operations.

---

## Content Ops

For libraries, CMS, ingestion pipelines, content quality systems, and search-and-recovery tools.

References:

- Shopify Polaris
- GitHub Primer
- Notion / Airtable-like content organization patterns
- IBM Carbon (for diagnostic sections)

Borrow:

- Readable resource lists.
- Search and filter clarity.
- Task queues.
- Warm but efficient surfaces.
- Content quality and status signals.

Avoid:

- Pure DevOps coldness when users think in terms of content rather than infrastructure.

---

## Public Marketing / Consumer Delight

For websites, landing pages, consumer onboarding, and brand storytelling.

References:

- Apple HIG principles for polish and motion.
- Vercel / Linear / Framer-style marketing.
- Product-specific industry references.

Borrow:

- Strong brand memory.
- Controlled, purposeful motion.
- Simple hierarchy with clear conversion paths.

Avoid:

- Applying marketing hero patterns to operational tools.
